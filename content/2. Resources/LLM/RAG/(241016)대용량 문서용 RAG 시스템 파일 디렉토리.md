# 대용량 문서용 RAG 시스템 파일 디렉토리

> 저자: Chia Jeng Yang (WhyHow.AI 공동 창립자)
> 출처: Medium · Knowledge Graph RAG · 2024년 10월 16일

## 요약

수만 건 문서를 다루는 엔터프라이즈 RAG에서 파일 디렉토리 구조로 검색 공간을 먼저 좁히는 2단계 RAG 설계.

## 핵심 포인트

- 2단계 RAG: 1단계(디렉토리 필터링) → 2단계(벡터 검색)
	- 관련 문서군을 먼저 좁힌 뒤 검색
- 문서 수가 많아질수록 단일 벡터 검색은 노이즈 증가
	- 사전 필터링이 정확도 유지의 핵심
- 파일 디렉토리 구조 자체가 하나의 지식 계층으로 작동

## 인사이트

엔터프라이즈 규모에서는 "좋은 임베딩 모델"보다 "검색 공간을 어떻게 설계하느냐"가 더 중요할 수 있다. 도서관에서 책을 찾을 때 전체를 뒤지는 게 아니라 분류 체계를 따라가는 것과 같은 원리.

---

## 핵심 내용

### 문제 정의

엔터프라이즈 RAG 시스템에서 수만 건의 문서를 다룰 때, **쿼리와 가장 관련성 높은 문서를 어떻게 효율적으로 찾을 것인가?**

기존 RAG는 모든 문서를 벡터로 검색하지만, 문서 수가 많아질수록 **검색 공간이 넓어지고 정확도가 떨어진다.**

---

### 핵심 아이디어: 2단계 RAG

```
[1단계] 어떤 문서가 관련 있는가?
        File Directory Agent → Knowledge Graph of Documents
                                        ↓ (문서 목록 반환)
[2단계] 해당 문서에서 답을 찾아라
        Knowledge Graph of Specific Document → Final Answer
```

**1단계** — 쿼리에 관련된 문서 목록을 지식 그래프에서 추출
**2단계** — 필터링된 문서 각각의 전용 그래프에서 정보를 추출해 최종 답변 생성

---

### 구현 흐름

![[구현흐름.webp]]

#### Step 1. 지식 테이블(Knowledge Table) 생성

각 문서에 대해 사전 정의된 질문을 LLM으로 실행해 메타데이터 생성:

| 질문 | 예시 답변 (s23_manual.pdf) |
|------|--------------------------|
| 이 제품의 설명은 무엇입니까? | Samsung Galaxy S23 |
| 모델에 대한 설명은 무엇입니까? | Galaxy S23 / S23+ / S23 Ultra |
| 이 제품은 어떤 카테고리에 속합니까? | Phone |
| 이 제품에 대한 설명은 무엇입니까? | Must return |

→ 셀 값이 특정 문서와 연결되는 **메타데이터 스키마**를 구성

#### Step 2. 메타데이터 → 지식 그래프 변환

```python
# WhyHow SDK로 Manual Graph 생성
from whyhow import WhyHow

client = WhyHow(api_key="<whyhow-api-key>", base_url="https://api.whyhow.ai")
# 트리플(Triple)로 내보내어 문서 그래프 생성
```

두 가지 그래프 생성:
- **Manual Graph** (전체 디렉토리): 모든 문서의 메타데이터 연결망 → 관련 문서 목록 반환
- **Product-specific Graph** (개별 문서): 특정 문서 내 상세 정보 → 최종 답변 추출

#### Step 3. 2단계 에이전트 쿼리

```python
def get_docs(q: str):
    """1단계: Manual Graph에서 관련 문서 목록 반환"""
    query = "이 질문에 답하려면 어떤 파일을 봐야 하나요?: " + q
    query_response = client.graphs.query_unstructured(
        graph_id="<manual-graph-id>",
        query=query
    )
    return query_response

def query_specific_graph(graph: str, query: str):
    """2단계: 개별 문서 그래프에서 정보 추출"""
    answer = client.graphs.query_unstructured(
        graph_id="<graph-id>",
        query=query
    )
    return answer.answer

def get_final_answer(answers: list, query: str):
    """최종 답변 통합 (GPT-4o-mini)"""
    prompt = f"""
    Answer the following query using the information provided in the data below.
    Explain your answer.
    Query: {query}
    Data: {answers}
    """
    response = openai_client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        max_tokens=512,
    )
    return response.choices[0].message.content
```

#### Step 4. 전체 실행 예시

**쿼리:** *"2023년에 출시된 기기 중 생체 인식 기능을 갖춘 제품은?"*

```python
query = "Which 2023 devices have biometric capabilities?"

# 1단계: 관련 문서 탐색
manuals = get_docs(query)
# → s23_manual.pdf, notebook_manual.pdf, washer_manual.pdf 반환

# 2단계: 각 문서 그래프 쿼리
relevant_docs = []
for node in manuals.nodes:
    if node["label"] == "Product":
        doc_property = node['properties'].get('document')
        relevant_docs.append(doc_property)

answers = []
for doc in relevant_docs:
    answer = query_specific_graph(doc, query)
    answers.append({"answer": answer, "source": node["name"]})

# 최종 답변
final = get_final_answer(answers, query)
# → "The 2023 devices with biometric capabilities are the
#     Samsung Galaxy S23 and the Samsung notebook computer."
```

### 메타데이터 유형에 따른 다른 결과

같은 시스템이라도 **쿼리 방식에 따라 다른 결과**를 얻을 수 있음:

| 쿼리 | 검색 기준 | 반환 문서 | 최종 답변 |
|------|----------|----------|----------|
| "2023년 생체인식 기기는?" | 연도(2023) | S23, 노트북, 세탁기 매뉴얼 | S23, 노트북 |
| "폰의 생체인식 기능은?" | 카테고리(Phone) | S21, S22, S23 매뉴얼 | 각 모델별 지문·얼굴 인식 상세 답변 |

### 확장 가능성

이 패턴은 **문서 유형에 무관하게** 적용 가능:
- 제품 카탈로그
- 유지보수 매뉴얼
- 학술 논문
- 법률 문서
- 의료 기록

Knowledge Table의 질문 스키마만 바꾸면 된다.

## 인상적인 부분

- **2단계 접근법**이 핵심: 벡터 검색으로 모든 문서를 뒤지는 대신, **먼저 관련 문서를 추려서** 검색 공간을 대폭 줄임 → 정확도·속도 동시 향상
- UI 없이 **순수 코드로** 대규모 문서 스트리밍 및 메타데이터 생성 가능 — 엔터프라이즈 자동화에 적합
- 같은 쿼리라도 메타데이터 기준(연도 vs 카테고리)에 따라 **완전히 다른 답변**이 나온다는 점 — 지식 그래프의 유연성

## 내 생각 / 적용 아이디어

- 기존 RAG 파이프라인의 병목은 "어떤 청크를 가져올 것인가"인데, 이 방식은 **문서 단위 필터링을 선행**해 그 문제를 구조적으로 해결
- 삼성 제품 카탈로그 예시처럼, 사내 문서(보고서, 규정, 매뉴얼 등)에 적용하면 엔터프라이즈 Q&A 봇 구축 가능
- WhyHow KG Studio가 오픈소스이므로 직접 실험해볼 가치 있음
