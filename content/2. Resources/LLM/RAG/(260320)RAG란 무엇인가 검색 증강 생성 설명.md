# RAG(Retrieval-Augmented Generation)란 무엇인가요?

> 저자: Paolo Perrone (Data Science Collective 창립 편집자)
> 출처: Medium, 2026-03-20
> 부제: 법학 석사 과정 학생이 사실을 지어내는 것을 막는 방법

## 요약

에어 캐나다 챗봇 패소 사건으로 시작해 RAG의 개념·파이프라인·5가지 한계를 실전 사례와 함께 설명.

## 핵심 포인트

- RAG = "AI의 오픈북 시험"
	- LLM이 답을 기억하는 게 아니라 문서에서 읽어오는 구조
- 역설: 생성(Generation)보다 검색(Retrieval)이 더 중요
	- Garbage in, Garbage out
- 5가지 한계: 청킹 어려움 · Lost in the Middle · 멀티모달 처리 · 지식 기반 품질 · 평가 부재
	- 현재 운영 중인 RAG 시스템의 70%가 평가 없이 운영

## 인사이트

60%의 기업 AI가 RAG 기반인데 70%는 평가 없이 운영된다는 숫자가 무섭다. RAG를 구축하는 것보다 RAG가 제대로 작동하는지 평가하는 것이 더 어렵고, 결국 그게 품질을 결정한다.

## 도입: 에어 캐나다 챗봇 사건

에어 캐나다 챗봇이 슬픔에 잠긴 승객에게 **이미 폐지된 장례 할인 정책**을 자신 있게 안내. 승객은 예약 후 환불 신청했지만 거절당함. 재판부는 에어 캐나다에 배상금 지급 명령. 챗봇은 웹사이트에서 삭제.

> AI 챗봇은 이를 방지하도록 설계된 환경에서도 **3~27%의 확률**로 잘못된 정보를 제공한다. 현재 실제 운영 중인 AI 애플리케이션의 **60%**는 기업 데이터를 기반으로 질문에 답변하는 시스템을 사용.

## RAG란?

> "RAG는 AI를 위한 오픈북 시험이다."

- LLM처럼 기억에 의존해서 답을 내놓는 대신, 시스템은 먼저 관련 문서를 검색한 후 실제로 찾아낸 내용을 바탕으로 답을 생성한다
- **핵심 통찰**: LLM의 역할은 답을 *아는* 것이 아니라, 올바른 문서에서 답을 *읽는* 것
  - 검색기(Retriever): 소싱 담당
  - 생성기(Generator): 커뮤니케이션 담당

## "모델에게 직접 물어보세요" 전략의 5가지 실패

### 문제 1: 오래된 정보
모델은 이전 정책을 기반으로 학습되었기 때문에 정책 변경을 알지 못한다. 자신 있게 답변하고, 고객은 신뢰하고, 결국 소송에서 패소.

### 문제 2: 출처 불명확
모델이 유창한 답변을 생성하지만 **출처를 확인할 방법이 없다**. 최신 정책 문서인지, 이전 문서인지, 블로그 게시물인지, 학습 데이터 패턴인지 알 수 없다.

### 문제 3: 불확실성 하에서 오판
모델은 정보가 부족할 때 "모르겠습니다"라고 말하지 않고, 보간법을 사용해 그럴듯한 텍스트를 생성:
- 스탠포드 연구: 법률 전문 AI 도구조차 **17~34%** 확률로 잘못된 정보 생성
- 일반 모델: 법률 관련 질의에서 최대 **82%** 오판

### 문제 4: 재학습 없이는 업데이트 불가
정책 변경, 문서 업데이트, 새 제품 추가 때마다 재학습 또는 미세조정 필요 → 비용 많이 들고, 시간 오래 걸리며, 빠르게 변하는 정보에 비현실적.

### 문제 5: 규모의 경제
지식 기반에 10만 개의 문서가 있다면, 각 질문에 관련성 있는 3~5개의 문서를 찾는 것이 모든 문서를 프롬프트에 집어넣는 것보다 훨씬 효율적.

> ⚠️ **"컨텍스트 창이 엄청나게 커지고 있는데, 모든 걸 다 넣으면 안 될까요?"**
> Gemini 같은 모델은 100만 개 이상의 토큰을 지원하지만, 장기 컨텍스트 모델은 여전히 대규모 환경에서 "건초 더미에서 바늘 찾기" 문제를 해결하지 못하고, 모든 쿼리에서 실행 비용이 높으며, 답변이 어떤 문서에서 나온 것인지 알려줄 수 없다. **RAG와 장기 컨텍스트는 경쟁 관계가 아니라 상호 보완적 관계**.

## RAG 작동 방식: 4단계

![[0_JxglRO78g7WY421s.webp]]

```
[오프라인 - 데이터 수집 파이프라인]
Knowledge Base → Chunk → Embed → Vector Store
                                      ↕
[런타임 - 쿼리 파이프라인]
User Query → Embed → Retrieve(Top-K) → Augment → Generate → Grounded Answer
              Query→Vector  관련 덩어리  Chunks+Query→Prompt  LLM 읽고 생성  + 출처 인용
```

### 1단계: 질문을 벡터로 변환
자연어 질문 → 임베딩 모델이 벡터(숫자 리스트)로 변환. 표현이 완전히 달라도 의미가 유사하면 같은 결과 검색 가능.

### 2단계: 관련 문서 덩어리 검색
쿼리 벡터를 벡터 저장소의 모든 문서 덩어리와 비교 → 의미적으로 가장 유사한 상위 K개 검색. *(학생이 교과서에서 관련 페이지를 찾아보는 것과 같음)*

### 3단계: 프롬프트 보강
검색된 조각들 + 원래 질문 = 하나의 증강된 프롬프트

```
Based on the following context, answer the user's question.

Context:
[Bereavement policy doc, last updated January 2026]
"Bereavement fares must be requested PRIOR to ticket purchase.
Retroactive applications are not accepted..."

[FAQ: Fare types]
"Bereavement fares are available for immediate family members.
Documentation required within 14 days of travel..."

Question: Can I apply for a bereavement fare after I've already booked my flight?
```

### 4단계: 근거 있는 답변 생성
LLM이 증강된 프롬프트를 받아 답변 생성. 이제 실제 현재 정책이 컨텍스트에 있다. 답변은 **출처 인용과 함께** 반환됨.

## 실제 사례

### Harvey AI (법률)
- RAG 기반 법률 연구 플랫폼 구축
- 45개국 판례법·법령·규정을 포함한 3단계 데이터 처리 (1~50개 파일 / 1,000~10,000개 문서 / 서드파티 법률 DB)
- PwC와 협업한 Tax AI Assistant: 기성 ChatGPT보다 **91% 더 선호**
- 결론: 기본 검색은 법률 영역 단순 질문 답변 가능, 실제 판례 연구는 커스텀 훈련 모델이 필요. **"RAG는 기반이지 천장이 아니다"**

### Shopify (커머스)
- Sidekick: help docs, store data, product catalogs 기반 RAG
- 판매자가 "제 고객 중 토론토 출신은 몇 명인가요?"와 같은 자연어 질문 가능
- **Just-in-Time Instructions**: 모든 도구 지침을 시스템 프롬프트에 넣으면 성능 저하 → 각 특정 작업에 관련된 컨텍스트만 로드 → RAG 원칙 그대로 적용

### Hugging Face (벤치마크)
- 156개 쿼리, 핵 공학 문서 대상 테스트
- 놀라운 결과: **Naive chunking이 context-aware chunking보다 우수** (70.5% vs 63.8%)
- 청크 크기는 거의 영향 없음. 하이브리드 검색이 dense-only에 패배
- **교훈: "모범 사례"를 맹목적으로 따르지 말 것. 자신의 문서 유형으로 벤치마크할 것**

## 잘못될 수 있는 것 (5가지 한계)

### 1. Garbage in, garbage out (더 감지하기 어려운)
지식 기반에 오래되거나 모순되거나 잘못 작성된 문서가 있으면 RAG가 충실하게 검색하고 인용한다. 시스템은 문서가 **정확한지**가 아니라 **관련성이 있는지**만 평가한다. 에어 캐나다 챗봇은 오래된 정책 문서를 완벽하게 검색했다. **검색은 완벽했고, 문제는 지식 기반이었다.**

### 2. 청킹은 미해결 예술
문서를 청크로 분할하는 방법이 검색 품질에 극적인 영향을 미치며, 보편적인 답이 없다:
- 너무 작게 분할 → 문맥 손실
- 너무 크게 분할 → 관련성 희석
- 잘못된 경계에서 분할 → 단락의 의미 파괴
- 올바른 청킹 전략은 **도메인 의존적**이며 종종 직관에 반한다

### 3. "Lost in the Middle" 문제
연구에 따르면 LLM은 컨텍스트 창의 **처음과 끝**에 있는 정보에 더 주의를 기울인다. 중요한 청크가 8개 검색 문서 중 4번째 위치에 있으면 모델이 가중치를 낮출 수 있다. **검색 순서가 대부분의 팀이 인식하는 것보다 더 중요하다.**

### 4. 평가가 여전히 주관적
2025년 기준, **70%의 RAG 시스템에 체계적인 평가 프레임워크가 없다**. 팀들은 개선 사항을 배포하고 나아졌기를 바란다. RAGAS, Braintrust 같은 도구들이 이 문제를 해결하기 시작했지만, 분야는 아직 미성숙. 검색 품질을 측정할 수 없다면 개선할 수 없다.

### 5. "RAG is dead" 과대광고 점검
2026년 초 "RAG is dead" 주장 → 대부분 낚시성 기사. 1M+ 토큰 컨텍스트 창 확장으로 인한 것이지만:
- 소규모 지식 기반: 장기 컨텍스트 모델이 RAG 필요성 감소
- **수천 개 문서, 빠르게 변하는 데이터, 비용 민감한 배포, 감사 요구사항에는 RAG 유효**
- Agentic RAG, GraphRAG, 하이브리드 검색으로 진화 중
- "답변 전에 찾아보기" 패턴은 근본적

> **공학 교훈**: RAG와 메모리는 같은 것이 아니다. RAG는 쿼리 시점에 문서를 검색한다. 메모리란 세션 간에 사용자의 선호도를 기억하고, 과거 상호 작용에서 학습하며, 자체 이해를 업데이트하는 에이전트를 의미한다. 하나는 검색이고, 다른 하나는 상태다.

## 꼭 기억해야 할 단 한 가지

> "RAG는 모델을 더 똑똑하게 만드는 것이 아니다. RAG는 모델의 **정보 소비 방식**을 바꾼다."

**"학습 과정에서 암기한 모든 것"** → **"지금 이 질문과 실제로 관련된 특정 문서"**

모델 자체는 동일하지만 입력값이 완전히 달라진다. 바로 이 차이가 소송으로 이어지는 챗봇과 고객의 신뢰를 얻는 챗봇을 가르는 핵심이다.

## 시사점

- RAG의 핵심은 **검색 > 생성**: 성공 여부를 결정짓는 것은 문서를 어떻게 묶고, 어떻게 검색하며, 검색 결과에 올바른 맥락이 나타나는지
- 청킹 전략이 전체 시스템 성능에 가장 큰 영향 — 도메인별 벤치마크 필수
- "Lost in the Middle" 문제: 검색 순서와 결과 배치가 LLM 성능에 영향
- 지식 기반 품질 관리가 RAG 시스템의 신뢰성을 결정
- 장기 컨텍스트 모델과 RAG는 경쟁이 아닌 상호 보완 관계
- RAG ≠ 메모리: RAG는 검색, 메모리는 상태 유지
