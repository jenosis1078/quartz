# 당신의 첫 번째 RAG: 기초 튜토리얼

> 저자: 스칸다 비벡 (EMAlpha)
> 출처: Medium · 2024-09-18

## 요약

RAG의 6단계 파이프라인(추출→청킹→임베딩→저장→검색→생성)을 Amazon 재무 보고서 실습으로 처음부터 끝까지 안내.

## 핵심 포인트

- 청킹 전략이 RAG 품질을 좌우
	- 고정 크기 vs 의미 단위 분할의 트레이드오프, 도메인마다 최적값 다름
- 임베딩은 텍스트를 벡터로 변환, 코사인 유사도로 의미 기반 검색 가능
- LangChain + OpenAI + FAISS 조합이 빠른 프로토타이핑의 기본 스택

## 인사이트

RAG를 "검색 + 생성"으로 단순하게 이해했는데, 실제로는 청킹 전략 하나로도 품질이 크게 달라진다. 맹목적 모범 사례보다 도메인에 맞는 청킹 전략을 직접 실험해야 한다는 점이 핵심.

---

## 개요

RAG(Retrieval Augmented Generation)는 LLM이 학습 데이터에 없는 외부 지식을 활용할 수 있도록 검색 기능을 결합한 기술이다. 
이 글은 RAG의 기초 개념부터 실습까지 6단계 파이프라인을 소개한다.

## RAG 6단계 파이프라인


![[일반적인 RAG 적용 사례(스칸다 비벡).webp]]

```
문서 추출 → 청킹 → 임베딩 → DB 저장 → 관련 청크 검색 → LLM 출력
```

### 1단계: 문서 추출 (Document Extraction)
- PDF, HTML, TXT 등 다양한 포맷의 문서를 텍스트로 변환
- 실습 예시: Amazon Q1-2023 10-Q 재무 보고서

### 2단계: 청킹 (Chunking)
긴 문서를 LLM이 처리 가능한 작은 단위로 분할하는 과정

| 방법 | 설명 | 특징 |
|------|------|------|
| 고정 크기 청킹 | 일정한 토큰/문자 수로 분할 | 단순하지만 문맥 단절 위험 |
| 재귀적 청킹 | 구분자(단락→문장→단어) 순으로 분할 | 문맥 보존 우수 |
| 문서 기반 청킹 | 문서 구조(제목, 섹션)에 따라 분할 | 구조화된 문서에 적합 |
| 의미적 청킹 | 의미 유사성 기반 분할 | 고품질이지만 복잡 |
| 에이전트 기반 청킹 | LLM이 직접 청킹 결정 | 가장 정교하지만 비용 높음 |

### 3단계: 임베딩 (Embedding)

![[벡터 검색 101(스칸다 비벡).webp]]

![[벡터 검색 기초(단어활용)(스칸다 비벡).webp]]

- 텍스트 청크를 고차원 벡터로 변환
- 모델 예시: `OpenAI text-embedding-ada-002`
- 의미적으로 유사한 텍스트는 벡터 공간에서 가까운 위치에 배치

### 4단계: 벡터 DB 저장 (Vector Database Storage)
- 임베딩 벡터를 벡터 데이터베이스에 저장
- 빠른 유사도 검색(ANN, Approximate Nearest Neighbor) 지원
- 도구 예시: Chroma, Pinecone, Weaviate, FAISS

### 5단계: 관련 청크 검색 (Relevant Chunk Retrieval)

![[검색 및 생성.webp]]

사용자 쿼리와 가장 관련성 높은 청크를 검색하는 두 가지 방법:

**키워드 검색 (Sparse Retrieval)**
- TF-IDF: 문서 내 단어 빈도 × 전체 문서 내 희소성
- BM25: TF-IDF 개선 버전, 문서 길이 정규화 포함
- 장점: 빠르고 해석 가능 / 단점: 동의어, 문맥 파악 불가

**벡터 임베딩 검색 (Dense Retrieval)**
- 쿼리와 청크를 각각 임베딩 후 코사인 유사도 계산
- 코사인 유사도 = 두 벡터의 내적 / (벡터1 크기 × 벡터2 크기)
- 장점: 의미적 유사성 파악 / 단점: 계산 비용 높음

### 6단계: LLM 출력 (LLM Generation)
- 검색된 청크를 컨텍스트로 LLM 프롬프트에 삽입
- LLM이 컨텍스트 기반으로 답변 생성
- 환각(Hallucination) 감소 효과

## 실습 예시

**목표**: Amazon Q1-2023 10-Q 문서로 재무 QA 시스템 구축

```python
# 1. 문서 로드
from langchain.document_loaders import PyPDFLoader
loader = PyPDFLoader("amazon-10q-q1-2023.pdf")
documents = loader.load()

# 2. 청킹
from langchain.text_splitter import RecursiveCharacterTextSplitter
splitter = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=200)
chunks = splitter.split_documents(documents)

# 3. 임베딩 & DB 저장
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
embeddings = OpenAIEmbeddings()
vectordb = Chroma.from_documents(chunks, embeddings)

# 4. 검색 & 생성
from langchain.chains import RetrievalQA
from langchain.llms import OpenAI
qa_chain = RetrievalQA.from_chain_type(
    llm=OpenAI(),
    retriever=vectordb.as_retriever()
)
result = qa_chain.run("Q1 2023 Amazon 순이익은?")
```

## 한계 및 주의사항

- **테이블 내 데이터 추출 부정확**: PDF 테이블은 텍스트 추출 시 구조 손실 발생
- **청크 크기 트레이드오프**: 너무 작으면 문맥 손실, 너무 크면 노이즈 증가
- **임베딩 모델 의존성**: 임베딩 모델 품질이 검색 성능에 직접 영향
- **비용**: 임베딩 생성 및 LLM API 호출 비용 고려 필요

## 시사점

- RAG는 LLM의 지식 한계를 극복하는 실용적인 접근법
- 도메인 특화 문서(법률, 의료, 재무 등)에 특히 효과적
- 프로덕션 환경에서는 하이브리드 검색(키워드 + 벡터) 조합 권장
- 청킹 전략이 전체 시스템 성능에 가장 큰 영향을 미침
