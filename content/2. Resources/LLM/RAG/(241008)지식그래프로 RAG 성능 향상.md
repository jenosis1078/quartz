# 지식 그래프를 활용하여 RAG 애플리케이션의 성능을 향상시키세요

![[1_1YlXoCGOLJs-CnTYw-f31Q.webp]]

## 개요

LLM 환각(Hallucination) 문제를 해결하는 세 가지 접근법 중 RAG가 가장 정교한 솔루션으로 평가된다. 그 중에서도 **지식 그래프(Knowledge Graph)**를 RAG에 결합하면 검색 정확도와 관계 이해를 크게 향상시킬 수 있다.

> 원문은 2024-10-07 singlestore.com에 게시, 이후 Medium Level Up Coding에 재게시

## 요약

기존 벡터 검색 RAG의 한계를 지식 그래프와 결합하여 극복 — 엔티티 간 관계 정보까지 활용하는 KG-RAG 구현.

## 핵심 포인트

- 벡터 DB는 의미 유사도 검색에 강하지만 관계 정보를 보존하지 못함
	- "A가 B의 상위 개념" 같은 계층적 관계 소실
- 지식 그래프: 엔티티(노드) + 관계(엣지)로 구조화
	- 다단계 관계 추론 가능
- SingleStore로 벡터 검색과 그래프 탐색을 동시에 처리하는 하이브리드 방식

## 인사이트

일반 RAG의 약점은 "관계"다. "CEO가 누구냐"는 쉽지만, "이 CEO가 이전에 근무한 회사의 경쟁사는?"처럼 다단계 관계 추론이 필요하면 벡터 검색만으로는 한계가 있다. KG-RAG가 복잡한 질문에 강한 이유다.

---

## LLM 환각 해결을 위한 3가지 접근법

1. **RAG (Retrieval Augmented Generation)** - 외부 지식 검색 결합 (가장 정교)
2. **미세 조정 (Fine-tuning)** - 특정 도메인 데이터로 재학습
3. **프롬프트 엔지니어링** - 지시문 최적화

## 지식 그래프란?

![[1_jsxZJojMXcv3wZQ5wT8j4g.webp]]

지식 그래프는 **엔티티(노드)** 간의 **관계(엣지)**를 그래프 구조로 표현하는 방식이다.

**예시 텍스트**: "소와 개는 동물의 좋은 예입니다. 소는 풀을 먹는데, 풀은 식물입니다!"

→ 추출되는 관계:
- 소 **is** 동물
- 개 **is** 동물
- 소 **eat** 허브(풀)
- 허브 **is** 식물
- 식물/동물 **is** 살아있는 생물

이런 비정형 텍스트에서 구조화된 지식 그래프를 자동 생성 가능하다 (LLM 활용).

## RAG 기본 구조

RAG 파이프라인의 3가지 핵심 요소:

![[0_zcBWeMdXkc9HbWzl.webp]]

| 요소 | 역할 |
|------|------|
| **검색 (Retrieval)** | 벡터 DB에서 관련 정보 조회 |
| **증강 (Augmentation)** | 검색 결과로 컨텍스트 보강 |
| **생성 (Generation)** | LLM이 컨텍스트 기반 답변 생성 |

## 지식 그래프가 RAG를 향상시키는 방법

### 핵심 이점
- **의미론적 맥락 제공**: 엔티티 간 관계와 속성을 이해하여 더 정확한 검색
- **관련 없는 결과 필터링**: 오해 소지 있는 결과 제거로 출력 품질 향상
- **정형/비정형 데이터 통합**: 두 데이터 유형을 연결하여 풍부한 맥락 제공
- **다단계 질문 처리**: 연결된 정보 조각들을 탐색하는 복잡한 질의 해결

### 지식 그래프 RAG 작동 방식

![[0_UMAMITGdsMylqnEn.webp]]

```
1. 데이터 수집     → 관련 데이터셋 식별 및 수집
2. 데이터 모델링   → 노드(엔티티) 및 엣지(관계) 유형 스키마 정의
3. 그래프 구성     → 엔티티/관계 생성, 데이터 정규화
4. 데이터 수집     → SQL로 DB에 데이터 적재
5. 인덱싱 및 검색 → 하이브리드 검색 전략 구현
```

### 프레임워크 및 도구
- **LangChain**: 엔티티 인식 및 관계 매핑 자동화로 지식 그래프 구축 간소화
- **LLM**: OpenAI, Meta, Google, Anthropic, Microsoft 등의 LLM으로 지식 그래프 생성
- **그래프 표준**: RDF, OWL, 그래프 데이터베이스 (Neo4j 등)
- **벡터 DB**: SingleStore (하이브리드 검색 지원)

## 벡터 DB RAG vs 지식 그래프 RAG 비교

![[0_NQK8nSsB_kpxiLFX.webp]]


```
벡터 DB RAG:                    지식 그래프 RAG:
Input Query                     Input Query
    ↓                               ↓
Vector Search                   Knowledge Graph Search
    ↓                               ↓
Retrieve Relevant Docs          Retrieve Relevant Entities
    ↓                             and Relationships
Generate Response                   ↓
                                Generate Response
```

| 항목 | 벡터 DB RAG | 지식 그래프 RAG |
|------|------------|----------------|
| 검색 방식 | 의미적 유사도(임베딩) | 구조화된 관계 탐색 |
| 강점 | 대규모 비정형 데이터, 빠른 설정 | 엔티티 관계 중요 도메인, 다단계 질의 |
| 적합 상황 | 관계가 명확하지 않은 경우 | 엔티티 간 연결 심층 이해 필요 시 |
| 구현 복잡도 | 낮음 | 높음 (스키마 설계 필요) |

> **참고**: 두 방식 모두 SingleStore에서 통합 가능 (벡터 + 그래프 하이브리드)

## 실습: SingleStore로 지식 그래프 RAG 구현


![[0_bMULWs4Dkkmknjlq.webp]]

**예제 데이터**: 슈퍼 마리오 브라더스 텍스트/데이터셋

### 1단계: 스키마 생성

```sql
-- 엔티티 테이블
CREATE TABLE entity (
    id INT,
    name VARCHAR(255),
    type VARCHAR(50),
    FULLTEXT (name)
);

-- 관계 테이블
CREATE TABLE relationship (
    id INT,
    from_entity_id INT,
    to_entity_id INT,
    type VARCHAR(50),
    FULLTEXT (type)
);
```

### 2단계: 엔티티 데이터 삽입

```sql
INSERT INTO entity (id, name, type) VALUES
(1, 'Super Mario Bros.', 'game'),
(2, 'Nintendo', 'company'),
(3, 'Famicom', 'console'),
(4, 'Nintendo Entertainment System', 'console'),
(5, 'Mario Bros.', 'game'),
(6, 'Super Mario series', 'series'),
(7, 'Mushroom Kingdom', 'location'),
(8, 'Princess Toadstool', 'character'),
(9, 'King Koopa', 'character'),
(10, 'Bowser', 'character'),
(11, 'Super Mushroom', 'item');

INSERT INTO relationship (id, from_entity_id, to_entity_id, type) VALUES
(1, 1, 2, 'developed_by'),
(2, 1, 2, 'published_by'),
(3, 1, 3, 'released_on'),
(4, 1, 4, 'released_on'),
(5, 1, 5, 'successor_to'),
(6, 1, 7, 'part_of'),
(7, 1, 8, 'set_in'),
(8, 1, 9, 'features'),
(9, 1, 9, 'features'),
(10, 1, 10, 'same_as'),
(11, 1, 11, 'contains');
```

### 3단계: 관계 쿼리 (전체 텍스트 검색)

```sql
-- "Princess Toadstool"과 관련된 모든 관계 검색
SELECT relationship.*,
       entity_from.name AS from_entity_name,
       entity_to.name AS to_entity_name,
       0.5 * IFNULL(MATCH(entity_from.name) AGAINST('Princess Toadstool'), 0) +
       0.5 * IFNULL(MATCH(entity_to.name) AGAINST('Princess Toadstool'), 0) AS score
FROM relationship
JOIN entity AS entity_from ON relationship.from_entity_id = entity_from.id
JOIN entity AS entity_to ON relationship.to_entity_id = entity_to.id
WHERE
    MATCH(entity_from.name) AGAINST('Princess Toadstool') OR
    MATCH(entity_to.name) AGAINST('Princess Toadstool')
ORDER BY score DESC;

-- 단순 엔티티 검색
SELECT * FROM entity WHERE MATCH(name) AGAINST ('princess');
-- 결과: Princess Toadstool (character)
```

### 아키텍처 흐름

```
Input Text Data
    ↓
LLM (OpenAI): Generate the KG  →  Knowledge Graph
                                        ↓
                                   SingleStore
                                   (Search Query)
                                        ↑↓
                              User inputs question → User
                                        ↓
                              LLM (OpenAI): Generate NL response
                                        ↓
                              Natural language response
```

## 지식 그래프 구축 모범 사례

1. **명확한 스키마 정의**: 표현할 엔티티와 관계 유형을 사전에 정의
2. **고품질 데이터**: 정보의 관련성과 정확성 보장
3. **쿼리 언어 학습**: Cypher(Neo4j) 등 그래프 쿼리 언어 습득
4. **지속적 업데이트**: 정확성과 관련성 유지를 위한 정기적 갱신

## 일반적인 문제 및 해결책

| 문제 | 해결책 |
|------|--------|
| 문맥 왜곡 (쿼리 ≠ 결과) | 지식 그래프로 관련 없는 결과 필터링 |
| 다단계 질문 답변 어려움 | 노드-관계 네트워크로 연결 문서 탐색 |
| 정형/비정형 데이터 통합 | 지식 그래프가 두 유형 연결 |

## 응용 분야

- **의료**: 환자 데이터 통합, 동물 품종-질병-치료법 연계
- **금융**: 사기 탐지
- **전자상거래**: 개인 맞춤형 추천 엔진
- **고객 지원**: 챗봇, 질문 답변 시스템

## 시사점

- 지식 그래프 RAG는 단순 벡터 검색보다 **관계 중심 도메인**에서 강력
- SQL DB(SingleStore)로도 그래프 저장/쿼리 가능 (JOIN, 재귀 CTE 활용)
- 전용 그래프 DB가 반드시 필요한 것은 아님
- LLM이 텍스트에서 자동으로 지식 그래프 생성 가능 → 구축 비용 절감
