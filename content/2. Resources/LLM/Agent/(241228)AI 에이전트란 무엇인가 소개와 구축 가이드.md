# AI 에이전트란 무엇인가? 소개와 구축 가이드

## 개요

가트너는 AI 에이전트가 미래의 핵심 기술이라고 예측한다. OpenAI, 엔비디아, 마이크로소프트, 세일즈포스까지 AI 에이전트에 투자하고 있다. 이 글은 AI 에이전트의 개념과 실제 구축 방법을 보험 청구 처리 사례로 설명한다.

> 저자: Maximilian Vogel (BIG PICTURE Co-founder, ML/GenAI 전문가)

## 요약

보험 청구 처리 사례를 통해 AI 에이전트의 개념과 4단계 구축 방법을 초보자도 이해할 수 있게 설명.

## 핵심 포인트

- 에이전트 = 바리스타(판단+실행), 자동화 = 에스프레소 머신(고정 순서)
	- 스스로 무엇을, 어떻게 할지 결정
- 구축 4단계: 목표 정의 → LLM 선택 → 도구 연결 → 오케스트레이션 로직 설계
- Gartner 예측: 2028년 기업 에이전트 사용률 33%
	- 아직 초기 단계

## 인사이트

"에스프레소 머신 vs 바리스타" 비유가 에이전트의 본질을 잘 포착한다. 자동화는 정해진 순서를 빠르게 실행하지만, 에이전트는 상황을 판단하고 순서 자체를 결정한다. 이 차이가 작아 보여도 구현 복잡도는 완전히 다르다.

## 핵심 개념: 에이전시(Agency)

기존 생성형 AI와 달리, AI 에이전트는 **복잡한 문제를 처음부터 끝까지 자율적으로** 처리할 수 있다.

**비유: 에스프레소 머신 vs 바리스타**

| | 에스프레소 머신 (전통 AI) | 바리스타 (AI 에이전트) |
|--|--|--|
| 역할 | 커피만 만듦 | 손님 맞이, 주문, 서빙, 계산, 문 닫기까지 |
| 특징 | 단일 작업 수행 | 전체 프로세스 자율 운영 |

> **"AI 에이전트는 마법이 아니다. 인간 직원이 수행하는 동일한 단계를 그저 100배 빠르게 실행한다."**

## Traditional GenAI vs Agentic GenAI 비교

| 특성 | Traditional GenAI | Agentic GenAI |
|------|------------------|---------------|
| 처리 방식 | 직접 답변 생성 (단순 흐름) | 다단계 문제 처리 (복잡한 흐름) |
| 자율성 | Dependent - 인간이 프롬프트 필요 | **Autonomous** - 내/외부 트리거로 자체 시작 |
| 능력 | Can answer - 질문 답변, 콘텐츠 생성 | **Can reason and decide** - 추론 후 다음 단계 결정 |
| 메모리 | Stateless - 세션 내 단기 메모리 | **Stateful** - 수일/수주/수개월 컨텍스트 유지 |
| 결정론 | Non-deterministic - 확률 기반 | **Large deterministic parts** - 검증/처리를 위한 결정론적 체인 |

**효율성 차이**
- Traditional GenAI: **5~15%** 팀 효율 향상
- AI 에이전트: **100~200%** 효율 향상 (end-to-end 완전 처리)

## AI 에이전트 구축 4단계

실습 예시: 보험사 보험 청구 처리 에이전트

### 1단계: 작업 분류 및 처리 레인 라우팅 (Classification & Routing)

고객 메일 수신 → LLM이 메시지 유형 분류 → 해당 처리 레인으로 라우팅

```
메일 수신
    ↓
AI: classify MAIL_TYPE
    ↓
CLAIM → lane PROCESS_CLAIM → step extract_data_from_mail → ...
QUESTION → lane ANSWER_QUESTION → step classify_type_of_question → ...
CHANGE_CONTRACT → lane CHANGE_CONTRACT → step extract_data_from_mail → ...
```

**핵심**: 분류는 텍스트를 미리 정의된 범주에 할당하는 것 (추출과 다름)

### 2단계: 데이터 추출 (Data Extraction)

비정형 텍스트 → 구조화된 JSON 데이터로 변환

**입력 (비정형 텍스트)**:
```
Hi, I would like to report a damage...
Deepak Jamal
contract no: HC12-223873923
123 Main Street, 10008 New York City
(718) 123 45678
```

**출력 (구조화된 JSON)**:
```json
{
  "name": "Deepak",
  "surname": "Jamal",
  "address": "123 Main Street, 10008 New York City, NY",
  "phone": "+1 718 123 45678",
  "contract_no": "HC12-223873923",
  "claim_description": "Yesterday [Dec-8, 2024], while playing..."
}
```

- 텍스트뿐 아니라 이미지(손상 사진), PDF(청구서)도 처리
- LLM, 이미지 인식 모델, OCR 등 복수 모델 활용
- **모델 출력 형식을 JSON으로 강제**하여 구조화 데이터 보장

### 3단계: 외부 서비스 호출 & 컨텍스트 지속 (Call External Services / Persistent Context)

> "거의 모든 강력한 AI 에이전트는 기업 또는 외부 데이터베이스에 접근해야 한다."

- 계약번호를 고객 DB에서 검증 (유효/무효 분기 처리)
- 청구 상태를 이슈 트래킹 시스템에 기록
- 필요 시 고객에게 누락 데이터 요청 이메일 발송
- **세션을 넘어 프로세스 컨텍스트를 지속**시키는 것이 핵심

```
step check CONTRACT_NO
    ↓
Valid → lane INITALIZE_CLAIM → step create insurance claim → ISSUE TRACKING DB
Not valid → lane ANSWER_CLIENT (계약번호 오류 안내 메일 발송)
```

### 4단계: 평가, RAG, 추론 및 신뢰도 (Assessment / RAG / Reasoning / Confidence)

**RAG 활용**: 모든 약관 정보를 프롬프트에 넣을 수 없으므로 벡터 DB에서 관련 스니펫 검색

**추론 (Chain of Thought)**: AI에게 답변 전 "소리 내어 생각"하도록 프롬프트
- 고객에게 근거 설명 가능
- 오류 원인 분석 용이
- 정답 도달 경로의 적절성 검증

**신뢰도(Confidence) 기반 자동화 레벨 조정**:
```
신뢰도 임계값 설정
    ↓
신뢰도 높음 → 자동 처리 계속
신뢰도 낮음 → 휴먼 핸드오버 (사람에게 전달)
```

높은 임계값 = 오류 최소화, 수동 처리 증가
낮은 임계값 = 더 많은 자동화, 오류 위험 증가

## AI 에이전트의 3가지 법칙

*(Isaac Asimov 로봇 3원칙 패러디)*

1. **에이전트는 인간의 도움 없이 end-to-end로 작업을 완수해야 한다**
2. **에이전트는 처리가 안전하고 보안적임을 보장하는 규칙을 따라야 한다**
3. **에이전트는 최고의 정확도를 달성하기 위해 추론과 신뢰도를 활용해야 한다**

## 성과

위 4단계를 구현하면 보험 청구 관리 부서 워크로드의 **70~90%를 자동화** 가능. 단순 생성형 AI로는 불가능한 수준.

## 구현 도구

- **프레임워크**: crewAI, LangGraph, LangFlow
- **순수 코드**: Python으로도 구현 가능
- 2~3단계만 구현해도 에이전트라고 할 수 있음

## 시사점

- AI 에이전트의 핵심은 **에이전시(자율성)** - 단순 답변 생성을 넘어 전체 프로세스 처리
- **Stateful(상태 유지)**이 에이전트를 가능하게 하는 핵심 특성
- RAG + 추론 + 신뢰도의 조합으로 신뢰할 수 있는 자동화 구현
- 신뢰도 임계값으로 자동화 수준과 안전성의 균형 조절 가능
- 에이전트 = 인간이 하는 동일한 단계를 100배 빠르게 수행
