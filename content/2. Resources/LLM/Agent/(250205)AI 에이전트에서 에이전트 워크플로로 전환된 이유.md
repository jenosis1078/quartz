# 인공지능 에이전트에서 에이전트 기반 워크플로로 초점이 옮겨간 이유

> 저자: Cobus Greyling (Chief Evangelist @ Kore.ai)
> 출처: Medium, 2025-02-05

## 요약

단일 에이전트의 현실적 정확도 한계(14~50%)를 데이터로 보여주고, 이를 극복하기 위한 에이전트 워크플로 패러다임 전환을 설명.

## 핵심 포인트

- 에이전트 정확도 현실: ACI 14% / TheAgentFactory 20% / OpenAI Operator 30~50%
	- 최고 수준도 절반 이하
- 워크플로 분해 효과: 각 단계 검증 가능 + 실패 시 재시도 + 관찰 가능성 확보
- 트렌드 전환: 단일 에이전트 → 전문화된 에이전트들의 협업 워크플로

## 인사이트

14%라는 숫자가 충격적이다. 과대광고와 현실의 갭이 이렇게 크다. 단일 LLM에 모든 것을 맡기는 건 현재 기술 수준에서 비현실적이며, 워크플로 분해는 선택이 아닌 필수다. 2025년 2월 기준 이 상황인데 그 이후 얼마나 개선됐을지 지속 추적이 필요하다.

## 핵심 주장

AI 에이전트의 **정확도 한계**로 인해, 업계의 초점이 단일 에이전트에서 **에이전트 기반 워크플로우(Agentic Workflows)**로 이동하고 있다.

## AI 에이전트 정확도 현황

현실의 벤치마크 결과는 기대에 미치지 못한다:

![[0_0haiVYzqtGR8TKrU.webp]]

| 벤치마크 | 성공률 |
|----------|--------|
| ACI (일반 에이전트) | **14%** |
| TheAgentFactory | **20%** |
| OpenAI Operator | **30~50%** |

> 에이전트의 작업이 복잡할수록 성공률은 급격히 하락한다.

## 왜 단일 에이전트는 한계가 있는가?

- 복잡한 작업을 **하나의 LLM 호출**로 처리하려 할 때 오류 누적
- 중간 과정에 대한 **관찰 가능성(Observability)** 부족
- 실패 시 어느 단계에서 문제가 발생했는지 파악 어려움
- 긴 컨텍스트에서 LLM의 추론 능력 저하

## 에이전트 기반 워크플로우로의 전환

### 핵심 전략: 하위 작업 분해

복잡한 작업을 **더 작은 하위 작업(sub-task)**으로 나누어 처리:

```
복잡한 작업
    ↓
하위 작업 1 → 에이전트 A (전문화)
하위 작업 2 → 에이전트 B (전문화)
하위 작업 3 → 에이전트 C (전문화)
    ↓
결과 통합
```

**이점**:
- 각 단계별 검증 및 오류 감지 가능
- 전문화된 에이전트로 각 단계 최적화
- 실패 지점 명확히 파악하여 재시도 용이
- 관찰 가능성과 제어 가능성 향상

### Stairway to Human-Like Interactions

인간과 유사한 상호작용으로 향하는 계층 구조:

```
Level 5: 완전 자율 에이전트
Level 4: 에이전트 기반 워크플로우
Level 3: 도구 사용 LLM
Level 2: RAG 기반 응답
Level 1: 기본 LLM 응답
```

## Agentic RAG

LlamaIndex가 제시하는 개념으로, RAG를 단순 검색-생성이 아닌 **에이전트 루프** 안에 통합:

- **데이터 합성(Data Synthesis)**: 여러 데이터 소스에서 정보를 가져와 종합
- 에이전트가 필요한 데이터를 능동적으로 쿼리하고 결합
- 단순 검색 → 추론 기반 다단계 정보 통합으로 발전

```
쿼리
  ↓
에이전트 (추론)
  ↓ ↑ (반복)
검색 도구들 (여러 데이터 소스)
  ↓
종합된 응답
```

## Reasoning & Problem-Solving

### 질문 분해 전략

복잡한 질문을 단계적으로 분해하여 처리:

- **DeepSeek** 방식: 내부적으로 추론 체인을 생성 후 답변
- 질문을 하위 질문으로 분해 → 각각 처리 → 통합
- 불확실한 경우 명시적으로 "모름"을 인정하고 추가 정보 요청

### 에이전트 워크플로우 패턴들

| 패턴 | 설명 |
|------|------|
| ReAct | 추론(Reason) + 행동(Act) 반복 루프 |
| Plan & Solve | 먼저 계획 수립 후 단계별 실행 |
| REWOO | 계획 단계에서 도구 사용 분리 |
| LLM Compiler | 병렬 실행으로 효율 향상 |
| LATS | 트리 탐색 기반 에이전트 |

## 결론

> **"도구에 집착하지 말고, 실제 비즈니스 문제 해결에 집중하라."**

- 단순히 "에이전트를 만들었다"는 것보다 **실제로 무엇을 해결하는가**가 중요
- 에이전트 기반 워크플로우는 현실적인 정확도 한계를 극복하는 실용적 접근
- 관찰 가능성과 제어 가능성이 프로덕션 배포의 핵심 요건
- 완전 자율 에이전트보다 **인간 감독 하의 반자율 워크플로우**가 현재로서 더 현실적

## 시사점

- AI 에이전트의 정확도는 생각보다 낮다 — 과대광고와 현실의 갭 인식 필요
- 복잡한 작업은 단일 에이전트보다 **전문화된 에이전트들의 협업** 구조가 유리
- RAG도 에이전트 루프 안에 통합하면 단순 검색을 넘어 데이터 합성 가능
- 프로덕션에서는 각 단계의 **가시성(visibility)**과 **오류 복구 전략**이 필수
