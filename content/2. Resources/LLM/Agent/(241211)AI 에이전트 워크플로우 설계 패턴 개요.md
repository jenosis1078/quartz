# AI 에이전트 워크플로우 설계 패턴 - 개요

> 저자: Craig Li, Ph.D
> 출처: Medium · Binome · 2024-12-11

## 요약

AI 에이전트 워크플로우를 성찰 중심형과 계획 중심형으로 분류하고, ReAct·Plan&Solve·REWOO·LLM Compiler·LATS 5가지 패턴을 비교.

## 핵심 포인트

- ReAct: 추론(Reason)과 행동(Act)을 번갈아 반복
	- 가장 범용적, 단순 태스크에 효율적
- REWOO: 계획 먼저 수립 후 일괄 실행
	- 툴 호출 횟수 최소화, 병렬화 가능
- LLM Compiler: 태스크를 DAG로 분해하여 병렬 실행
	- 속도 최적화에 유리

## 인사이트

패턴 선택은 태스크의 복잡도와 툴 호출 비용에 따라 달라진다. 단순 반복 → ReAct, 계획이 먼저 필요 → Plan&Solve, 속도가 중요 → LLM Compiler. 하나의 패턴을 고집하지 말고 상황에 맞게 조합하는 것이 현실적이다.

## 개요

![[1_08KTKf0vmwm-pTdvhOcoSg.webp]]

LLM 기반 AI 에이전트는 **추론(Reasoning) + 메모리(Memory) + 도구 실행(Tool Use)**을 결합한 시스템이다. 워크플로우 설계 패턴은 에이전트가 문제를 해결하는 방식을 구조화하며, 크게 **성찰 중심형**과 **계획 중심형**으로 분류된다.

> **핵심 인사이트**: 모든 에이전트 설계 패턴의 본질은 **"인간의 사고와 관리 전략을 구조화된 프롬프트로 변환"**하는 것이다.

## 패턴 분류 체계

```
AI 에이전트 워크플로우 설계 패턴
├── 성찰 중심형 (Reflection-focused)
│   ├── 기본 성찰 (Basic Reflection)
│   ├── 고찰 (Reflexion) - 강화학습 기반
│   ├── LATS - 트리 탐색 + 성찰 + 강화학습
│   └── 자기 발견 (Self-Discover)
└── 계획 중심형 (Planning-focused)
    ├── Plan & Solve
    ├── LLM Compiler - 병렬 실행
    ├── REWOO - 관찰 없는 추론
    └── Storm - 대용량 문서 생성
```

## 핵심 기반: ReAct 패턴

![[1_08KTKf0vmwm-pTdvhOcoSg 1.webp]]

모든 패턴의 토대. 2022년 발표된 논문에서 소개.

**원리**: "언어적 추론에 의해 안내되는 행동"
- 행동 후 자기 반성: "내가 방금 무엇을 했나? 목표를 달성했나?"
- 이 관찰(Observation)이 에이전트의 단기 메모리 역할

**진화 과정**:
```
Reason Only          Act Only              ReAct
(Chain-of-Thought)   (SayCan, WebGPT)      (Reason + Act)
LM → Reasoning       LM ↔ Env              LM ↔ Env
     Traces          Actions/Observations   Actions/Observations
                                           + Reasoning Traces
```

### ReAct 5단계 구현 흐름

![[1_yw_98xQCBzzpuJ0nzLEYDw 1.webp]]


```
1. 프롬프트 생성
   Question → Thought → Action → Observation 형식 + few-shot 예시
        ↓
2. LLM: Thought + Action 생성
   (Stop 컨트롤로 Observation 생성 방지)
        ↓
3. 외부 도구 호출 (Action ≠ "Finish")
   Action → API 형식으로 변환 → 외부 도구 실행
        ↓
4. API 결과를 자연어 Observation으로 변환
        ↓
5. 최종 답변 생성 (Action = "Finish")
```

**예시** - 책상에서 펜 찾기:
```
Action 1: 펜꽂이 확인
Observation 1: 펜꽂이에 없음 → 다음 단계로
Action 2: 서랍 확인
Observation 2: 서랍에 펜 있음
Action 3: 서랍에서 펜 꺼내기 ← 목표 달성 시 종료
```

**ReAct 구현 커스터마이징 2가지**:
1. 프롬프트 템플릿의 few-shot 예시 (비즈니스 시나리오에 맞게)
2. 함수 호출을 위한 외부 도구 정의

![[1_WQarFPrWeatrSZHeRoHjTQ.webp]]

## 성찰 중심형 패턴

### 1. Basic Reflection (기본 성찰)

![[1_9543fUoJVC5xq9Ot5aLbvw.webp]]

**비유**: Generator(학생) ↔ Reflector(선생님) 피드백 루프

```
User Request
    ↓
Generate (생성자)
    ↓
Initial Response
    ↓
Reflect (반사자) ← N번 반복
    ↓
Reflections (비평/장점/개선안)
    ↓
다시 Generate → 만족스러울 때까지 반복
    ↓
Respond to user
```

### 2. Reflexion (고찰) - 강화 학습 기반

![[1_owP9OsMtoMBNT5BnGYR1GA.webp]]

Basic Reflection에 강화 학습 원리 추가.
- **외부 데이터**로 응답 평가
- 중복/누락 강제 해결 → 더 정교한 반성 과정

```
Responder → Initial Response (Response + Critique + Search)
    ↓
Execute Tools
    ↓
Revisor (N번 반복) → Revised Response
    ↓
Respond to user
```

### 3. LATS (Language Agent Tree Search)

![[1_IpwqQIk8i7k-lLf9HRQHSw.webp]]

**공식**: `LATS = 트리 탐색 + ReAct + Plan & Solve + 성찰 + 강화 학습`

여러 기법을 통합한 고급 패턴:
- 강화 학습 지식으로 트리 탐색 결과 평가
- 최적 결과를 위해 성찰 과정 통합
- 여러 생성자/반사자가 협력하는 다층 반사 구조

**반복 사이클**:
```
노드 선택 → 새 후보 생성 → Act, Reflect, Score → 역전파(Update parents)
```

### 4. Self-Discover (자기 발견)

![[1_dIw0pcymwtjUMVmYnwrBqw.webp]]

계획/해결이 "추가 단계나 조정이 필요한가"에 초점을 맞추는 반면, 자기 발견은 **작업 자체에 대한 성찰**을 장려한다.

**2단계 구조**:
```
Stage 1: Task-Level 추론 구조 발견
  Language Model + Task + Atomic Reasoning Modules
      → Self-Discover → Reasoning Structure (Key-Value 쌍)

Stage 2: Instance-Level 문제 해결
  Task Instance + Reasoning Structure → Language Model → Answer
```

## 계획 중심형 패턴

### 1. Plan and Solve (계획 후 실행)

- **ReAct 적합**: "책상에서 펜 찾기" (즉각적, 탐색적 작업)
- **Plan & Solve 적합**: "플랫 화이트 만들기" (사전 계획 필요, 계획 변경 가능)

```
User Request
    ↓
Planner (초기 계획 수립)
    ↓
Task List [1, 2, 3, ...]
    ↓
Single-Task Agent (Executor) → 도구 사용 루프
    ↓
Update state with task results
    ↓
Replanner (진행 상황 기반 재계획)
    ↓
5a. Respond to user (완료 시)
5b. Re-plan more tasks (미완료 시)
```

**주요 프롬프트 트리거 예시**:
| 타입 | 트리거 문장 |
|------|-----------|
| CoT | "Let's think step by step." |
| PS | "Let's first understand the problem and devise a plan..." |
| PS+ | "Let's first understand the problem, extract relevant variables... devise a plan... carry out the plan..." |

### 2. REWOO (Reason without Observation)

ReAct의 명시적 Observation 단계를 제거하여 프로세스 간소화.
- 이전 단계 결과가 **암묵적으로** 다음 실행 단위에 포함

```
ReAct:  Thought → Action → Observation → Thought → ...
REWOO:  Thought → Action → (암묵적 관찰) → 다음 Action → ...
```

**구성**:
- **Planner**: 의존성 포함 전체 태스크 리스트 생성
- **Worker**: 태스크 반복 실행 (도구 사용)
- **Solver**: 태스크 결과 통합 → 최종 응답

### 3. LLM Compiler (병렬 실행)

**컨셉**: 컴퓨터 과학의 컴파일러처럼 태스크 병렬 실행으로 효율 극대화

```
User Request → Planner
    ↓
Task DAG (방향성 비순환 그래프) 스트리밍
    ↓
Task Fetching Unit: A → B, C (병렬) → D (순차)
                    ↑ 의존성 $()로 표현
    ↓
병렬 스케줄링 + 실행 (as soon as possible)
    ↓
Joiner (재플래너)
    ↓
4a. Respond to user / 4b. Re-plan
```

**예시**: "AWS Glue와 MWAA의 차이점?" → 두 서비스 정의를 **동시에** 검색 후 결합

### 4. Storm (대용량 문서 생성)

위키피디아 유사 포괄적 문서를 처음부터 생성하는 워크플로우.

```
Topic → Related topics 검색
    ↓
Wiki editor personas + Expert persona
generate_question ↔ gen_answer (노드)
    ↓
Outline → Refined outline
    ↓
Section writer → Discussions (VectorStore)
    ↓
Formatter (Wiki Style) → Final Article
```

## 패턴 선택 가이드

| 상황 | 추천 패턴 |
|------|---------|
| 즉각적 탐색, 도구 사용 | **ReAct** |
| 사전 계획 필요, 동적 변경 | **Plan & Solve** |
| 병렬 처리로 효율 극대화 | **LLM Compiler** |
| 품질 개선, 반복 정제 | **Basic Reflection / Reflexion** |
| 복잡한 추론 + 최적 탐색 | **LATS** |
| 태스크 맞춤형 추론 구조 | **Self-Discover** |
| 대용량 구조화 문서 생성 | **Storm** |
| RAG 없이 단순 병렬 계획 | **REWOO** |

## 워크플로우 노드 유형

각 패턴에서 워크플로우의 노드는 다음을 나타낼 수 있다:
- **LLM 작업** (추론, 분류, 생성)
- **함수 호출** (외부 도구/API)
- **RAG 작업** (벡터 DB 검색)

## 시사점

- 에이전트 설계의 핵심은 **인간 사고 패턴을 프롬프트로 구조화**하는 것
- ReAct가 가장 보편적인 기반 패턴; 나머지는 이를 확장하거나 변형
- 실제 구현 시 커스터마이징 핵심 2가지: **few-shot 예시** + **도구 정의**
- Planner + Replanner 조합으로 동적 재계획 처리
- 복잡도가 높을수록 (LATS, Reflexion) 성능↑, 비용↑, 지연↑
