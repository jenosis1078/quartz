# AI 에이전트: 디지털 워커 구축하기

> 저자: Alfredo Sone
> 출처: Medium, 2024-11-18

## 요약

에이전트 설계를 직원 채용 과정에 빗대어 Perception·Brain·Memory·Knowledge·Tools 5요소와 Multi-Agent 패턴을 실무적으로 설명.

## 핵심 포인트

- 에이전트 5요소: Perception · Brain · Memory · Knowledge · Tools
	- Perception(입력 인식), Brain(LLM 추론·계획), Memory(단기/장기), Knowledge(지식 기반), Tools(API/도구)
- 설계 5단계: 목표 정의 → 도구 선택 → 메모리 설계 → 지식 기반 구축 → 오케스트레이션
- Multi-Agent 패턴 5종: Single / Network / Supervisor / Hierarchical / Custom

## 인사이트

"에이전트를 채용하듯 설계하라"는 비유가 실용적이다. 어떤 역할인지, 어떤 도구가 필요한지, 무엇을 기억해야 하는지를 먼저 정의하면 설계가 명확해진다. 무작정 LLM + 도구부터 붙이는 것이 아니라 역할 정의가 먼저다.

---

## 개요

AI 에이전트의 부상은 소프트웨어를 만들고 생각하는 방식을 재정의하고 있다. 단순히 컴퓨터에게 무엇을 할지 지시하는 것을 넘어, 이제는 **어떻게 생각할지**를 가르친다.

> "AI 에이전트는 LLM과 자동화를 결합하여 명령을 따르는 것이 아니라 추론하고, 의사결정하고, 행동하며, 심지어 시간이 지남에 따라 학습하는 도구를 만든다."

## AI 에이전트란?

전통적인 프로그램이 고정된 명령어를 따르는 것과 달리, 에이전트는 LLM과 지능을 사용하여 무엇을 해야 하는지, 어떻게 해야 하는지를 스스로 파악하고 실행한다.

### 업계 리더들의 정의

| 출처 | 정의 |
|------|------|
| **LangChain** | "LLM이 애플리케이션의 제어 흐름을 결정하는 시스템" |
| **NVIDIA** | "LLM을 사용해 문제를 추론하고, 해결 계획을 수립하고, 도구를 활용해 계획을 실행하는 시스템" |
| **AWS** | "환경과 상호작용하고, 데이터를 수집하며, 사전에 정해진 목표를 달성하기 위해 자기 결정적 작업을 수행하는 소프트웨어 프로그램" |

### LLM에서 에이전트로의 진화

```
텍스트 생성 (초기 LLM)
    ↓
오케스트레이션 (여러 LLM 협업)
    ↓
도구 호출 (API 사용)
    ↓
긴 컨텍스트 (더 많은 정보 처리)
    ↓
AI 에이전트 (자율적 행동)
```

**LLM은 수동 응답자에서 지능형 시스템으로 진화** — 고객 서비스, 채용, 법률, 건축 등 인간이 담당하던 역할로 확장 중.

## 에이전트의 해부학: 5가지 핵심 구성요소

![[1_zT_kZQg5-IP34aWrp1Vl_w.webp]]

```
┌─────────────────────────────────────────────┐
│                   Agent                      │
│  ┌─────────────────────────────────────────┐ │
│  │   Perception (환경과 상호작용)           │ │
│  └─────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────┐ │
│  │   Brain (추론, 계획, 적응)              │ │
│  └─────────────────────────────────────────┘ │
│  ┌──────────────┬──────────────┬───────────┐ │
│  │  Knowledge   │   Memory     │   Tools   │ │
│  │ (전문 지식)  │  (정보 유지) │ (행동 실행)│ │
│  └──────────────┴──────────────┴───────────┘ │
└─────────────────────────────────────────────┘
```

### 1. Perception (인식)

환경으로부터 입력을 받아 맥락을 파악:
- 입력 형태: 텍스트, 음성, 이미지, API 호출
- 목적: 주변 환경과 맥락을 이해하여 행동을 안내

> *예시: 항공 여행 에이전트가 "10월 5~10일 뉴욕 왕복 항공편 예약"이라는 요청에서 날짜, 목적지, 선호도를 추출*

### 2. Brain (두뇌)

LLM이 담당하는 추론·계획·적응 엔진:
- **Reasoning (추론)**: 입력 분석, 복잡한 작업 분해, 잠재적 해결책 생성
- **Planning (계획)**: 시간에 걸쳐 행동을 순서화하여 효율적으로 작업 완료
- **Adaptability (적응)**: 동적 환경이나 변화하는 정보에 대응

> *예시: 항공편 요청 수신 후 출발편·귀국편 찾기, 호텔 검색 등 단계 식별 및 필요 정보 정의*

### 3. Memory (메모리)

| 유형 | 저장 방식 | 용도 |
|------|-----------|------|
| **단기 메모리** | LLM 컨텍스트 윈도우 | 현재 세션 내 최근 상호작용 추적 |
| **장기 메모리** | 외부 데이터베이스 | 세션을 넘어 과거 대화, 사용자 선호도 저장 |

> *예시: 여행 에이전트가 세션 중 선택한 날짜와 클래스를 단기 메모리로 추적하고, 선호 항공사나 마일리지 정보는 장기 메모리로 유지*

### 4. Knowledge (지식)

회사별 지식 기반(FAQ, 문서, SOP)에서 정보를 가져와 정확한 응답 제공:
- 문서 및 벡터 데이터베이스에 저장
- **RAG 기법**으로 검색하여 추론과 의사결정 향상

> *예시: 고객이 수하물 정책이나 업그레이드 옵션을 물으면 회사별 정책을 검색하여 정확히 답변*

### 5. Actions/Tools (도구)

에이전트가 실제 행동을 취하게 하는 것:
- API를 통해 데이터 쿼리, 레코드 업데이트, 예약 수행
- LLM이 언제, 어떻게 도구를 사용할지 동적으로 결정
- 기존 시스템과 원활하게 통합

> *예시: 항공사 API로 실시간 항공편·가격·좌석 조회. 고객 승인 후 예약 API 호출 및 이메일 발송*

## 에이전트 설계: 직원 채용과의 유사성

> "효과적인 AI 에이전트를 설계하는 것은 새 직원을 채용하고 온보딩하는 것과 매우 유사하다. 우리는 본질적으로 **워커(일꾼)를 만들고 있다**."

### 5단계 설계 프로세스

#### 1단계: 페르소나와 목표 정의
에이전트의 "직무 기술서":
- 어떤 문제를 해결하는가?
- 사용자는 누구인가?
- 성공은 어떤 모습인가?

**팁**: 페르소나를 측정 가능한 결과물에 연결하고, 비즈니스 니즈에 따라 주기적으로 재검토

#### 2단계: 작업 및 계획 수립
에이전트의 "플레이북":
- 핵심 작업은 무엇인가?
- 엣지 케이스는 무엇인가?
- 따라야 할 규칙은?

**팁**: 복잡성과 일관성의 균형 — 흐름이 복잡하거나 작업이 많을수록 실패 가능성 증가

*Google Agents의 Goal + Instructions 구조:*
```
Goal: You are a shirt store ordering agent. Help customers purchase shirts...
Instructions:
- If the customer has not provided a size, ask them to choose a size.
- If the customer has not provided a color, ask them to choose a color.
- Once the customer has selected, repeat the selection and ask to confirm.
- Do not provide prices or collect payment.
```

#### 3단계: 메모리 정의
- 세션 내 무엇을 유지할 것인가?
- 시간이 지나도 어떤 데이터를 지속할 것인가?

![[0_12jorE-iLh8mrj34.webp]]


**팁**: 가장 가치 있는 데이터(예: 자주 접근하는 고객 정보)부터 시작하여 점진적으로 확장

#### 4단계: 지식 정의
- 에이전트에 필요한 지식 소스는?
- 어떻게 최신 상태를 유지할 것인가?

**팁**: 벡터 데이터베이스나 RAG 기법을 활용하고 정기적 업데이트 일정 수립

#### 5단계: 도구 장착
에이전트에게 직원에게 소프트웨어와 시스템을 제공하듯이:
- 필요한 API는 무엇인가?
- 보안을 위한 권한 수준은?

![[0_Q8zkhVaa1KZ78JTZ.webp]]


**팁**: 목적별 도구를 정의하고 역할 기반 권한 구현 (직원에게 적절한 소프트웨어와 접근 레벨 부여)

*OpenAI Function Calling 흐름:*
```
1. 앱이 프롬프트 + 함수 정의를 LLM에 전달
2. LLM이 사용자에게 응답할지, 함수를 호출할지 결정
3. API가 호출할 함수와 인자를 앱에 반환
4. 앱이 해당 인자로 함수 실행
5. 앱이 함수 결과를 포함하여 API 재호출
```

## 에이전트 팀 설계 (Multi-Agent)

복잡한 문제는 혼자 해결하기 어렵다. 하나의 대형 범용 에이전트 대신, **전문화된 에이전트들의 팀**이 더 효과적으로 작업을 처리한다.

### Multi-Agent 아키텍처 패턴 (LangChain)

| 패턴 | 설명 |
|------|------|
| **Single Agent** | LLM + 도구 세트 |
| **Network** | 에이전트들이 서로 통신하며 협력 |
| **Supervisor** | 상위 에이전트가 하위 에이전트 조율 |
| **Supervisor (as tools)** | 감독자가 도구로서 하위 에이전트 호출 |
| **Hierarchical** | 다단계 계층 구조 |
| **Custom** | 특정 요구에 맞춘 혼합 구조 |


![[0_jJYcQd2FMc2eqpVC 2.webp]]

**이점**:
- 각 에이전트가 더 작은 범위에 집중 → 신뢰성 향상
- 처리할 수 없는 작업을 다른 에이전트에 위임
- 협력 네트워크로 복잡한 문제 해결 = AI 인력(workforce) 구성

## POC를 넘어: 프로덕션 배포

### 반복적 개선
에이전트 구축은 본질적으로 반복적:
- 첫 번째 버전이 최종 버전이 된 적 없음
- 실제 조건에서 엣지 케이스와 숨겨진 가정 발견
- 사용자 평가, 오류 로그 수집 → 지속 개선

### 보안 (Security)
직원과 마찬가지로 에이전트에도 안전장치 필요:
- **SSO, 역할 기반 권한, 자격증명 관리**
- 가드레일, 로깅, 감사, 자격증명 로테이션
- 민감한 데이터 보호 및 컴플라이언스 유지

### 운영 (Operations)
COO가 워크플로우를 추적·최적화하듯:
- 에이전트도 감사 및 성능 추적 도구 필요
- 시간이 지나도 효과적이고, 신뢰할 수 있으며, 적응 가능하도록 유지

## 결론

> "AI 에이전트는 단순한 자동화 도구가 아니다 — 그들은 **디지털 협업자**다."

- AI 에이전트는 엄격한 규칙 기반에서 동적 추론 기반 시스템으로 자동화를 혁신
- 자율성은 강력하지만 신뢰성, 보안, 운영 관리의 도전도 가져옴
- 에이전트 설계 = 직원 채용과 온보딩: 페르소나, 작업, 메모리, 지식, 도구

## 시사점

- 에이전트의 핵심은 **Perception → Brain → Memory/Knowledge/Tools** 의 유기적 통합
- 단일 에이전트의 한계 → Multi-Agent 팀으로 복잡성 관리
- 프로덕션 배포에서 보안과 운영 관리는 선택이 아닌 필수
- 에이전트는 첫 버전이 완성본이 아님 — 반복적 개선 프로세스 내재화 필요
- Function Calling이 에이전트의 도구 사용을 가능하게 하는 핵심 메커니즘
