# 딥러닝 라이브러리 비교 (아키텍처 관점)

## 개념 요약

딥러닝 프로젝트 아키텍처 설계 시 라이브러리 선택 기준 비교표.

| 라이브러리 | 핵심 아키텍처 | 스케일링 방식 | 배포 아키텍처 | 컨테이너화 지원 | 장점 | 단점 | 프로젝트 적합도 |
|-----------|-------------|-------------|-------------|--------------|------|------|--------------|
| **PyTorch** | Dynamic Graph (Eager) | Single→Multi GPU (DDP), Distributed, Cloud | TorchServe, REST, gRPC, Edge(ExecuTorch) | Docker ✅ / K8s ✅ | 연구 유연성, 디버깅 용이 | 프로덕션 설정 복잡 | 연구5 / 프로토5 / 생산4 / 대규모4 |
| **TensorFlow** | Static/Dynamic Graph | Single→Multi GPU (MirroredStrategy), TPU, Cloud | TF Serving, REST, gRPC, TFLite(Edge) | Docker ✅ / K8s ✅ | 프로덕션 성숙도, 생태계 | 복잡한 API, 무거움 | 연구3 / 프로토3 / 생산5 / 대규모5 |
| **Keras** | High-level API (TF 백엔드) | TF 스케일링 그대로 상속 | TF Serving 통해 배포 | Docker ✅ / K8s ✅ | 빠른 프로토타이핑, 직관적 | 커스터마이징 한계 | 연구3 / 프로토5 / 생산3 / 대규모3 |
| **JAX** | Functional, XLA JIT | Multi-Device, TPU 최적화, pmap | 직접 구현 필요 (Flask 등) | Docker ✅ / K8s ⚠️ | 속도, TPU 최적화, 함수형 | 생태계 미성숙, 배포 도구 부족 | 연구5 / 프로토3 / 생산2 / 대규모3 |
| **MXNet** | Hybrid (Dynamic+Static) | Multi-GPU, Distributed, AWS 최적화 | MMS, REST, gRPC | Docker ✅ / K8s ✅ | AWS 통합, 경량 | 커뮤니티 축소, 개발 둔화 | 연구2 / 프로토2 / 생산3 / 대규모3 |
| **NumPy** | 배열 연산 (CPU only) | 단일 CPU | 직접 구현 필요 | Docker ✅ / K8s ⚠️ | 단순, 범용, 의존성 최소 | GPU 미지원, 딥러닝 직접 구현 불가 | 연구2 / 프로토2 / 생산1 / 대규모1 |
| **Pandas** | 데이터 전처리 특화 | 단일 머신 (Dask 조합 시 분산) | 전처리 파이프라인 용도 | Docker ✅ / K8s ⚠️ | 데이터 조작 최강, 직관적 | 대용량 메모리 한계, ML 직접 불가 | 연구3 / 프로토3 / 생산2 / 대규모2 |
| **Scikit-learn** | 전통 ML 알고리즘 | 단일 CPU (joblib 병렬) | REST API (Flask/FastAPI 래핑) | Docker ✅ / K8s ✅ | 전통 ML 완성도, 파이프라인 | 딥러닝 미지원, GPU 미지원 | 연구4 / 프로토4 / 생산4 / 대규모2 |

> **적합도 점수**: 1(부적합) \~ 5(최적)

---

## 수식

$\text{스케일링 효율} = \frac{\text{분산 처리 속도}}{\text{단일 GPU 속도} \times N}$

이상적인 선형 스케일링 = 1.0 / 실제 PyTorch DDP ≈ 0.9 / TF MirroredStrategy ≈ 0.85

---

## 직관적 설명
```
연구 아이디어 검증
└── PyTorch or JAX (빠른 실험, 디버깅)

프로토타입 → 서비스화
└── Keras (빠른 구현) → TensorFlow Serving (배포)

대규모 프로덕션
└── TensorFlow + K8s + TF Serving

엣지 디바이스 배포
└── TFLite (TF) or ExecuTorch (PyTorch)

전통 ML + 딥러닝 혼합
└── Scikit-learn (전처리/전통ML) + PyTorch (딥러닝)
```

## 구현 (코드)
```python
# 아키텍처 선택 가이드 (의사코드)
def select_framework(purpose, scale, deployment):
    if purpose == "research":
        return "PyTorch"  # 유연성 최우선
    elif purpose == "prototype":
        return "Keras"    # 빠른 구현
    elif scale == "large" and deployment == "cloud":
        return "TensorFlow"  # 성숙한 프로덕션
    elif deployment == "edge":
        return "TFLite or ExecuTorch"
    elif purpose == "traditional_ml":
        return "Scikit-learn"
```

> [!architecture] 아키텍처 선택 기준
> - **연구/프로토타입**: PyTorch (유연성), Keras (빠른 구현)
> - **중소규모 서비스**: PyTorch + TorchServe, FastAPI 래핑
> - **대규모 배포**: TensorFlow + TF Serving + Kubernetes
> - **엣지 디바이스**: TFLite (Android/iOS), ExecuTorch (PyTorch 2.0+)
